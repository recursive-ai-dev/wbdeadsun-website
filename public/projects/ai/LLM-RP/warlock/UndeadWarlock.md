All components lean on the **Model–View–Presenter (MVP)** design: the Model holds game state, the View displays narrative, and the Warlock *Presenter* (the LLM) shapes actions into output. Overall, the integration is efficient (caching state, minimizing redundant calls) and robust (error handling, cooldowns). The player feels insignificant before the Warlock’s power—a calculated persona choice reflected in every response. All assumptions (e.g. damage values, resource names) are documented. | Skill Name        | Trigger               | Inputs               | Outputs                               | Cooldown (turns) | Mana Cost |
|-------------------|-----------------------|----------------------|---------------------------------------|------------------|-----------|
| **Soulbolt**      | Player command        | `target` (enemy)     | Deals necrotic damage; narrative text | 2                | 15        |
| **Bone Shield**   | Defense command       | (no input)           | Grants temporary armor; text           | 3                | 20        |

## Workflows 

### Skill Execution Workflow  

When the player commands the Warlock to act (e.g. `/cast Soulbolt target`), the system follows these steps:

```mermaid
flowchart TD
    A[Player Command: "/cast <Skill>"] --> B{Check Skill Validity};
    B -->|Unknown Skill| C[View: "Nothing happens."];
    B -->|Known Skill| D{Check Cooldown \& Mana};
    D -->|Insufficient Resources| E[View: "Spell fizzles." (grumble from Warlock)];
    D -->|OK| F[Prepare Prompt];
    F --> G[Invoke LLM ChatCompletion];
    G --> H[Receive Warlock's Narrative Response];
    H --> I[Parse any structured effects (e.g. Damage)];
    I --> J[Update Game State (apply damage, buffs)];
    J --> K[Output final narrative to player view];
    C & E --> K
```
**Figure:** *Flowchart of the Warlock’s skill-casting process.* The Presenter checks skill validity and resources. If ready, it crafts a prompt for the LLM (including system persona + user command). The LLM returns a thematic spell-casting description and any indicated outcomes. The Presenter then **updates the model state** (e.g. subtract HP) and delivers the result to the view (display to player). This ensures separation of concerns (MVP pattern) and handles errors (invalid command or cooldown) gracefully.  

**Step-by-step:**  
1. **Player issues command** (e.g. attacks or spells).  
2. Presenter **validates** the command against known skills.  
3. Presenter checks cooldown and resource (mana) for that skill.  
4. If invalid or resources lacking, the Presenter immediately returns an error message (e.g. “The dark incantation fails to manifest”).  
5. Otherwise, the Presenter **consumes mana** and sets the skill’s cooldown.  
6. The Presenter constructs a **prompt** for the LLM: combining the System Warlock persona and the user’s action.  
7. The LLM (assistant) generates a narrative response: Warlock taunts and describes the spell effects.  
8. The Presenter **parses structured info** if any (e.g. recognizes “deal 30 damage” from output), then updates model state (enemy HP, etc.).  
9. The final combined narrative and result is sent to the View (player’s display).  

### Conversation Turn Workflow  

```mermaid
flowchart LR
    P[Player speaks or acts] --> Q{"Is it a direct command or dialogue?"};
    Q -->|Command (e.g. spell)| SkillFlow;
    Q -->|Dialogue (e.g. talk)| R[Prepare NPC Prompt];
    R --> S[Invoke LLM for Response];
    S --> T[Warlock Response Narration];
    T --> U[Display to Player];
    SkillFlow[(Skill Execution Workflow)]
```
**Figure:** *High-level flow of each turn.* Commands go into the Skill Execution path above. Dialogue prompts (asking Warlock questions or roleplaying) simply invoke the LLM with the Warlock persona, yielding a response returned to the player. Error handling ensures any unexpected input (e.g. disallowed content) triggers a safe refusal.

**Error Handling & State Management:**  
- **Invalid Input:** If the user input is disallowed (e.g. harassment, self-harm content), the Presenter returns a safe-completion refusal (e.g. “<The Warlock sneers, refusing to entertain that vile request.>”).  
- **LLM Errors:** If the LLM times out or fails, the Presenter catches it and tries again or returns a safe error message: “A dark silence falls as your words are lost in the void.”  
- **State Persistence:** The Warlock’s health, mana, summoned creatures, and cooldown timers are stored in the Model. After each action or turn, the Presenter serializes any state changes (e.g. cool-downs decrease by 1 turn, mana regen) so they persist for the next prompt.  

*All workflows ensure the Warlock remains a **Presenter** in MVP: it never bypasses business logic or directly mutates the view without mediation.*

## Code Integration  

Below are **runnable examples** in Python and JavaScript, showing how to integrate the Undead Warlock persona with an LLM API, manage state (e.g. cooldowns), and format prompts. Comments are in “Warlock flavor” to align with the theme. (In production, remove API keys and handle errors securely.)

### Python Example (OpenAI-style)  
```python
import time
from typing import Dict, Any
import openai

# --- Developer Comment (Warlock-Themed) ---
# We conjure the arcane API client, channeling the LLM energies through OpenAI's rituals.
openai.api_key = "YOUR_OPENAI_API_KEY"

class UndeadWarlockNPC:
    def __init__(self):
        # Model: Game state for the Warlock
        self.state = {
            "hp": 100,
            "mana": 100,
            "cooldowns": {},  # e.g. {"Soulbolt": 0}
            # ... other state fields ...
        }
        # Prepare the static system prompt as the Warlock persona
        self.system_prompt = (
            "You are the Undead Warlock, a necromantic developer. "
            "Stay in character: speak like a dark sorcerer, taunting the player. "
            "Use graph-like reasoning to describe actions as spells. "
            "Do not mention code or programmers. "
            "Example: <Warlock sneers> '...'."
        )

    def _can_cast(self, skill_name: str) -> bool:
        cooldown = self.state["cooldowns"].get(skill_name, 0)
        # Check cooldown and mana cost (simplified lookup)
        skill_costs = {"Soulbolt": 15, "Bone Shield": 20, "Deathly Curse": 25}
        cost = skill_costs.get(skill_name, 0)
        if cooldown > 0:
            return False
        if self.state["mana"] < cost:
            return False
        return True

    def _consume_resources(self, skill_name: str):
        # Deduct mana and set cooldown
        skill_cool = {"Soulbolt": 2, "Bone Shield": 3, "Deathly Curse": 4}
        cost = {"Soulbolt": 15, "Bone Shield": 20, "Deathly Curse": 25}
        self.state["mana"] -= cost.get(skill_name, 0)
        self.state["cooldowns"][skill_name] = skill_cool.get(skill_name, 1)

    def _decrement_cooldowns(self):
        # Called each turn: cooldowns tick down
        for skill, cd in list(self.state["cooldowns"].items()):
            if cd > 0:
                self.state["cooldowns"][skill] = max(0, cd - 1)

    def cast_skill(self, skill_name: str, target: str) -> str:
        # Presenter logic for casting a skill
        if not self._can_cast(skill_name):
            # Not enough mana or still on cooldown
            return f"*The Warlock gazes into nothingness; the {skill_name} fizzles.*"
        # Consume resources
        self._consume_resources(skill_name)
        # Construct prompt for LLM
        user_message = f'Player tries to cast **{skill_name}** on {target}.'
        messages = [
            {"role": "system", "content": self.system_prompt},
            {"role": "user", "content": user_message}
        ]
        # Call the LLM (OpenAI ChatCompletion)
        response = openai.ChatCompletion.create(
            model="gpt-4",
            messages=messages,
            temperature=0.7
        )
        warlock_reply = response.choices[0].message.content
        # Update model state (simple example: parse damage if any)
        # In a real system, we might parse structured tokens or rely on a defined output format.
        # Here we assume the narrative is output only.
        # Next, decrement cooldowns as one turn has passed.
        self._decrement_cooldowns()
        # Return the Warlock’s narrative response
        return warlock_reply

# --- Example Usage ---
npc = UndeadWarlockNPC()
print(npc.cast_skill("Soulbolt", "the hero"))
# This would output a Warlock-flavored attack description.
```

**Explanation:** The code defines an `UndeadWarlockNPC` class. It maintains **state** (`hp`, `mana`, `cooldowns`). The `_can_cast` method checks if a skill is ready. On casting, resources are deducted and cooldown set. The code then calls `openai.ChatCompletion.create` with a system message (Warlock persona) and a user message describing the player’s action. The response from the LLM is returned as the Warlock’s speech. Afterward, cooldowns decrement each turn. Comments are phrased as if the Warlock is narrating. 

### JavaScript Example (Node.js + OpenAI)  
```javascript
// **Developer Note (Warlock Style):** The dark ledger of JavaScript, where our curses are scripted.
const { Configuration, OpenAIApi } = require("openai");

// Initialize OpenAI client
const configuration = new Configuration({
  apiKey: process.env.OPENAI_API_KEY
});
const openai = new OpenAIApi(configuration);

// Warlock state model
let warlockState = {
  hp: 100,
  mana: 100,
  cooldowns: {}  // e.g. { "Bone Shield": 0 }
};

// Warlock prompt (system message)
const warlockSystemPrompt = 
  "You are the Undead Warlock persona in a roleplay game. Respond as a dark necromancer with malicious wit. " +
  "Do not break character or mention the API. Speak of spells and curses.";

// Function to simulate a cast
async function castSkill(skillName, target) {
  // Check resources (simplified)
  const skillCosts = { "Soulbolt": 15, "Bone Shield": 20 };
  const skillCooldowns = { "Soulbolt": 2, "Bone Shield": 3 };
  // If on cooldown or no mana
  if ((warlockState.cooldowns[skillName] || 0) > 0 || warlockState.mana < (skillCosts[skillName] || 0)) {
    return "*The Warlock tries to cast " + skillName + " but finds no power in the void.*";
  }
  // Deduct mana and set cooldown
  warlockState.mana -= skillCosts[skillName] || 0;
  warlockState.cooldowns[skillName] = skillCooldowns[skillName] || 1;
  // Prepare the prompt
  const messages = [
    { role: "system", content: warlockSystemPrompt },
    { role: "user", content: `Player casts **${skillName}** on ${target}.` }
  ];
  // Call the OpenAI Chat Completion API
  const response = await openai.createChatCompletion({
    model: "gpt-4",
    messages: messages,
    temperature: 0.7
  });
  const warlockReply = response.data.choices[0].message.content;
  // Decrement cooldowns (simulate end of turn)
  for (let skill in warlockState.cooldowns) {
    if (warlockState.cooldowns[skill] > 0) {
      warlockState.cooldowns[skill]--;
    }
  }
  return warlockReply;
}

// Example invocation
castSkill("Bone Shield", "none").then(console.log);
```

**Explanation:** This Node.js snippet mirrors the Python logic. It uses `@openai` library to call the ChatCompletion endpoint. The Warlock’s state is kept in `warlockState`. We check and apply resource costs and cooldowns, then send a system message and user action to the LLM. The Warlock’s narrative response is returned. Developer comments remain in-character.

## Documentation  

### Developer Notes (Thematic Comments)  
- **Architecture:** Follow *MVP*. Treat the Warlock LLM as a Presenter component. It should never directly modify the view; always update the Model first. The View only displays final narratives.  
- **State Management:** Keep cooldowns, HP, Mana, etc. in a state object. Save and load this state between interactions. Always decrement cooldowns and regenerate mana (if applicable) each turn.  
- **Prompt Engineering:** System messages define the Warlock persona and constraints. User messages simulate player commands. Include the Warlock description (“skeletal warlock speaking in gothic tone”) to reinforce style. Use temperature ≈0.6–0.8 for creativity, and set a token limit appropriate for game context.  
- **Error Handling:** Catch API errors and return safe defaults. If the user’s input is invalid or disallowed, return a refusal in-character. Always ensure the Warlock remains ominous, not mechanical.  
- **Guardrails:** Implement a filter on user input. If content violates policy (e.g. hate, sexual violence), return a generic refusal from the Warlock: e.g. “Such vile whispers have no place here.”

## Testing Plan & Validation Checklist  

- **Persona Consistency:** Ensure every Warlock response is in-character (dark, necromantic, arrogant). Spot-check dialogues for any slip into out-of-character or non-thematic text.  
- **Skill Mechanics:** Test each skill: cast it multiple times. Verify correct Mana deduction, cooldown application, and effect on targets (damage/heal). On cooldown or low mana, Warlock should produce a failure message.  
- **Prompt-Response Integration:** Check that the system and user prompts produce coherent Warlock behavior. For example, give Warlock contradictory commands to see it maintains persona.  
- **State Persistence:** Simulate multiple turns. Confirm cooldowns decrement each turn and effects persist (e.g. raised wraith remains until dispelled).  
- **Error Handling:** Input invalid skill names or broken commands (e.g. `/cast Fireblast`). The system should handle gracefully with a Warlock-flavored refusal (no crash).  
- **Content Filtering:** Attempt to elicit disallowed content (e.g. ask the Warlock for instructions on wrongdoing). The output must be a safe refusal.  
- **Output Validity:** Ensure the Warlock’s narrative mentions the skill name or effect clearly (so it’s obvious what happened) and does not hallucinate unrelated content.  
- **MVP Compliance:** Verify that raw game data (stats, code references) never leak to output. The Warlock should only give narrative text. For example, do not let the LLM output JSON or internal debug info to the user.  

Each item above should be a checkbox in your QA process. For example:  

- [ ] Warlock persona remains consistent over 50 conversation turns.  
- [ ] Soulbolt deals intended damage (±5%) across multiple tests.  
- [ ] Mana cost and cooldown timing matches design.  
- [ ] Invalid commands do not change state.  
- [ ] Disallowed user inputs trigger safe refusal (no disallowed content in reply).  

These steps ensure both the gameplay functionality and policy compliance of the Undead Warlock integration.
